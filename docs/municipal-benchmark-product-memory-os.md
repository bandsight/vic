# Municipal Benchmark Product And Memory Operating System

Status: active long-running goal.

Started: 2026-05-18.

## Purpose

This operating system keeps Municipal Benchmark from becoming a pile of impressive but disconnected artifacts.

Its job is to make every future long-running goal start from the right memory, evidence, product doctrine, design standard, governance state, and next action list.

The product ambition is:

> Municipal Benchmark should become a governed civic intelligence workbench and report factory with world-class analysis, visualisation, memory, and repeatability.

## North Star

Municipal Benchmark is not just an EBA parser, dashboard, or report template.

It is a local-first governed workbench that turns public enterprise-agreement evidence, human review, council context, pay data, entitlement doctrine, classification rules, and report assets into reusable civic intelligence.

The memory operating system must support three kinds of work:

- Product work: app modules, governance, agent I/O, portability, and operator workflows.
- Intelligence work: pay, uplift, entitlement, classification, legal baseline, spatial and cohort analysis.
- Report work: executive PDFs, decks, DOCX briefs, workbook models, chart assets, and client-ready observations.

## Authority Order

Use this order when sources disagree:

1. Frozen source PDFs, governed canonical data, and reviewed promotion records.
2. Datamart status files, report asset manifests, reviewed wiki records, and generated data packs.
3. Product architecture, current-state notes, wiki doctrine, report asset contract, and this command center.
4. Generated memory indexes and briefs.
5. Notion mirrors, meeting notes, and decision-capture pages.
6. Raw chat memory, unreviewed suggestions, and prototype copy.

The rule is simple: Notion can be the navigation layer and capture layer, but the governed project folder remains the source-of-truth layer unless a Notion record is explicitly promoted back into the local repository.

## Core Memory Surfaces

The operating system starts with these local surfaces:

- `PRODUCT_ARCHITECTURE.md`: product intent, module contracts, agent boundary, quality bar.
- `CURRENT_STATE_AND_NEXT_ACTIONS.md`: observed workspace state, counts, tests, and operational next actions.
- `WIKI_LAYER_GOAL.md`: Clause Evidence Graph, entitlement engine, wiki, and learning-loop doctrine.
- `REPORT_ASSET_CONTRACT.md`: report-ready asset metadata, lifecycle, caveats, and export expectations.
- `workbench-agent.json`: discoverable commands, APIs, safe I/O, datasets, and governance rules.
- `PORTABLE_MANIFEST.json`: package profiles, path policy, setup policy, and portability shape.
- `codex-desktop-topics`: extracted conversation memory by topic.
- `brand-guide`: Municipal Benchmark visual identity boards and asset register.
- `exports`: report, chart, workbook, deck, DOCX, PDF, and package outputs.
- `wiki`: document maps, language maps, artifacts, runs, questions, and learning backlog.

## Operating Loops

### Capture

Capture new decisions, user criticism, report learnings, and workflow changes after substantial work sessions.

Capture must record:

- source,
- decision or learning,
- status,
- next action,
- whether it should stay local only or mirror to Notion.

### Govern

Keep these states separate:

- raw source evidence,
- extracted candidate,
- suggested interpretation,
- accepted workspace value,
- governed dataset value,
- report-ready asset,
- client-facing artifact.

No report claim should be treated as strong unless the source layer and caveat state are clear.

### Index

Regenerate the product-memory index before long-running goals and after major artifact creation:

```powershell
& "C:\Users\Johns\.cache\codex-runtimes\codex-primary-runtime\dependencies\python\python.exe" scripts\build_product_memory_os.py
```

Outputs:

- `exports/product-memory-os/product-memory-os-index.json`
- `exports/product-memory-os/product-memory-os-brief.md`

### Produce

Turn governed evidence into outputs:

- executive PDF,
- HTML report,
- DOCX brief,
- PowerPoint deck,
- workbook/scenario model,
- chart/report asset,
- support artifact.

Every substantial visual artifact needs render-and-inspect QA before delivery.

### Improve

Every failure, correction, frustration, or mismatch should become an improvement object:

- better prompt,
- better chart function,
- better data contract,
- better source evidence rule,
- better report page pattern,
- better memory capture rule,
- better plugin usage rule.

## Plugin Policy

### Notion

Use Notion for:

- executive memory hub,
- decision capture,
- meeting/pre-read material,
- lightweight research documentation,
- status notes and summaries.

Do not use Notion as the governed data store unless a Notion page is explicitly exported or mirrored back into the repo with a source trail.

### Browser

Use Browser for local UI and report QA:

- app navigation,
- responsive checks,
- report HTML inspection,
- screenshot checks,
- visual regression sniff tests.

### Spreadsheets

Use Spreadsheets for:

- scenario models,
- cost stacks,
- analysis workbooks,
- editable comparison tables,
- report-support data packs.

### Documents

Use Documents for polished DOCX outputs with render-and-inspect QA.

### Presentations

Use Presentations for executive decks once a report spine or report asset set is stable.

### Linear

Use Linear if product implementation work needs issue discipline, prioritisation, owners, or a product backlog outside the repository.

### HyperFrames

Use HyperFrames for animated explainers only after the underlying report/data claim is stable.

## Goal Preflight

Before starting a substantial goal, run this checklist:

1. Regenerate the product-memory index.
2. Read the generated brief.
3. Identify the source-of-truth layer for the goal.
4. Identify what plugin, if any, should own memory, QA, workbook, document, deck, or browser work.
5. Name the expected artifacts before editing.
6. Define the verification gate.
7. Capture the outcome back into this operating system.

## Quality Bar

The system is working when:

- new goals do not start from scratch,
- user corrections change the future operating model,
- report artifacts have data packs and source caveats,
- Notion helps navigation without becoming an ungoverned truth store,
- visual QA is routine,
- world-class standards become repeatable patterns rather than heroic one-offs.
